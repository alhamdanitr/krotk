# Architecture Change Report (ACR-01)
**الموضوع:** تحويل النظام إلى منصة SaaS و Multi-Tenant

## 1. ملخص التعديل (Summary)
بناءً على طلب العميل، تم تغيير رؤية النظام بشكل جذري ليتحول من تطبيق فردي موجه لمستخدم واحد، إلى **منصة SaaS (Software as a Service)** تخدم آلاف العملاء (Tenants). يمتلك كل عميل بيئته المعزولة، إعداداته، ومخزونه الخاص، مع إضافة نظام إدارة اشتراكات وتفعيل آلي.

## 2. تأثير التعديلات على قاعدة البيانات (Database Impact)
تم تعديل ملف `schema.prisma` بالكامل وتحديث `DATABASE_SPECIFICATION.md`:
- **Logical Isolation:** استبدال `Shop` بـ `Tenant` وجعل `tenant_id` هو الأساس (Foreign Key) في جميع جداول النظام لعزل البيانات.
- **SaaS Tables:** إضافة جداول جديدة مخصصة لإدارة النظام:
  - `Plan`: إدارة الباقات (شهري، سنوي، الخ).
  - `ActivationKey`: مفاتيح التفعيل المكونة من `serialKey` و `activationCode`.
  - `Subscription`: سجل اشتراكات العملاء وتواريخ الصلاحية.
  - `User`: نظام مستخدمين يدعم الصلاحيات `role` لدعم الـ Super Admin ومدراء الـ Tenant.

## 3. تأثير التعديلات على الـ API (API Impact)
- جميع الـ Endpoints تم تعديلها لتتوقع `tenant_id` داخل الـ JWT Token.
- إضافة Endpoints جديدة لعملية التفعيل (Activation) وتسجيل المستأجر الجديد (Tenant Registration).
- لا يوجد أي Endpoint يسمح باستعلام بيانات دون فلترة بـ `tenant_id`، لحماية الخصوصية ومنع تسريب البيانات.

## 4. تأثير التعديلات على الـ Backend
- **Guard & Interceptors:** تمت إضافة خطط لتنفيذ `TenantGuard` لضمان أن كل مستخدم لا يصل إلا لبيانات الـ Tenant الخاص به.
- **Super Admin Preparedness:** هيكلة الكود تسمح مستقبلاً بإنشاء `SuperAdminModule` للتحكم في مفاتيح التفعيل وإيقاف الـ Tenants.

## 5. تأثير التعديلات على Flutter (الطرفيات)
- تطبيق الموبايل الحالي سيتحول إلى **Tenant Client App**. يجب على المستخدم إدخال `Serial Key` و `Activation Code` عند أول تشغيل ليتم التعرف على الـ Tenant الخاص به.
- إمكانية إضافة شاشة في الإعدادات تعرض حالة الاشتراك (تاريخ الانتهاء والباقة الحالية).

## 6. تأثير التعديلات على الواجهات (UI/UX - Disgain Folder)
- **تحليل واجهات Disgain (Jaib Fintech):** تم اعتمادها كمرجع بصري (Visual Reference) للنظام الجديد.
- **جودة التصميم:** سيتم استنساخ الجودة (Material 3، حواف 12px/16px، ألوان Slate الداكنة، والـ Primary Red).
- **إعادة التسمية:** سيتم تغيير الأسماء في مكونات الـ UI لتتناسب مع هوية **KUROTEK**، مثل استبدال أي مصطلحات مالية عامة بمصطلحات الكروت والشبكات، مع إزالة العناصر غير المتوافقة ودمج شاشات الإحصائيات (Home) والمخزون (Cards).

## 7. مخاطر وقرارات هندسية جديدة (Risks & Decisions)
- **الخطر 1 (تسريب البيانات):** إذا نسي المبرمج إضافة `WHERE tenant_id = ?` في أي استعلام بـ Prisma، سيرى العميل كروت عميل آخر.
  - *الحل الهندسي:* يمكن استخدام `Prisma Client Extensions` لإجبار فلترة الـ Tenant برمجياً على مستوى הـ ORM بدلاً من تركها لاجتهاد المبرمج.
- **الخطر 2 (تمدد قاعدة البيانات):** جميع العملاء يستخدمون نفس قاعدة البيانات `PostgreSQL`.
  - *الحل الهندسي:* تم تصميم الفهارس `Indexes` بدقة (مثلاً `@@index([tenantId, categoryValue])`) لضمان عدم تأثر الأداء حتى لو وصلت الجداول لملايين السجلات.
