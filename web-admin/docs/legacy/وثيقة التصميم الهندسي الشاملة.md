# Software Architecture Design (SDD) - Kurotek SaaS v2.5

## 1. المعمارية العامة (Global Architecture)
تعتمد المنصة على معمارية **Multi-Tenant SaaS** مع عزل منطقي صارم عبر `tenantId`. الخادم (NestJS) هو المحرك المركزي لجميع العمليات.

## 2. نظام الأمان والحماية (Double-Guard System)
جميع الـ APIs محمية بطبقتين إلزاميتين (Global Guards):
1. **JwtAuthGuard:** يتحقق من صحة الـ Access Token.
2. **TenantGuard:** يستخرج الـ Tenant من قاعدة البيانات ويحقنه في الـ Request.
3. **TrialGuard:** يتحقق من صلاحية فترة التجربة (7 أيام).
4. **SubscriptionGuard:** يتحقق من تاريخ انتهاء الاشتراك إن وُجد.
5. **RolesGuard:** يتحقق من صلاحيات الدور (`@Roles('super_admin')`).

المسارات العامة (Register, Login) تُعلَّم بـ `@Public()` لتجاوز هذه الحراسة.

## 3. هيكل قاعدة البيانات (Database Architecture - v2.5)

### 3.1 طبقة الإدارة (SaaS Layer)
- **`Plan`**: يحتوي على السعر، المدة، `maxUsers`، و`features` (JSON).
- **`ActivationKey`**: السيريال + كود التفعيل + `planId`.
- **`Subscription`**: تاريخ البدء والانتهاء وحالة الاشتراك لكل Tenant.

### 3.2 طبقة العميل (Tenant Layer)
- **`Tenant`**: الكيان الجذر. يحتوي الآن على: `name`, `logo`, `country`, `currency`, `status` (trial/active/suspended), `trialStartDate`, `trialEndDate`.
- **`User`**: مرتبط بـ `Tenant`، يحتوي على `role`.
- **`Settings`**: إعدادات الـ Tenant (SMS، صيغة الكرت).

### 3.3 طبقة العمليات (Business Layer)
- **`Category`**: فئات الكروت (500, 1000, 2000...) مرتبطة بالـ Tenant.
- **`Card`**: مخزون الكروت مع `isUsed` وفهرس مركب `[tenantId, categoryValue, isUsed]`.
- **`Deposit`**: الإيداعات الواردة من الـ SMS.
- **`Transaction`**: سجل الكروت المصرفة المربوط بالإيداع.
- **`PendingApproval`**: الإيداعات التي تحتاج مراجعة يدوية.
- **`CustomerMapping`**: خريطة الأرقام الوهمية للأرقام الحقيقية.
- **`WalletConfig`**: إعدادات الـ Regex لكل محفظة رقمية.
- **`SmsTemplate`**: قوالب الرسائل المرسلة للعملاء.

## 4. تجربة المستخدم (UX Flow)

### 4.1 Setup Wizard (5 خطوات - مرة واحدة فقط)
```
[شاشة الترحيب] → [اسم الشبكة + الشعار + الدولة + العملة] → [إنشاء الحساب] → [تفعيل التجربة] → [لوحة التحكم]
```

### 4.2 Splash/Auth Middleware (عند كل تشغيل)
```
فتح التطبيق → فحص JWT محلي → هل الـ Token صالح؟
  ├── نعم → فحص حالة الـ Tenant من الـ API
  │         ├── trial (أيام متبقية) → لوحة التحكم
  │         ├── active (اشتراك ساري) → لوحة التحكم
  │         └── expired / suspended → شاشة القفل
  └── لا  → شاشة تسجيل الدخول
```

### 4.3 Lockout Screen (شاشة القفل)
تظهر عند انتهاء التجربة أو الاشتراك وتحتوي على:
- رسالة واضحة (انتهت الفترة التجريبية / انتهى اشتراكك).
- تفاصيل الاشتراك المنتهي.
- **زر [إدخال السيريال]** → يفتح نموذج التفعيل.
- **زر [التواصل معنا]** → يفتح WhatsApp / صفحة الدعم.

## 5. هيكل Backend Modules (NestJS)

### مرحلة ١ و ٢ (مكتملة ✅)
- `PrismaModule` (Global)
- `AuthModule` (Login, Register Wizard, Activate Serial)
- Guards & Decorators

### مرحلة ٣ (قيد التنفيذ 🔜)
- `CardsModule`: CRUD لمخزون الكروت + Bulk Import.
- `SmsModule`: Webhook لاستقبال SMS + محرك Regex + سحب الكرت (Atomic Transaction).
- `DepositsModule`: تسجيل وعرض الإيداعات.

### مرحلة ٤+ (مستقبلية 🕓)
- `ReportsModule`: تقارير PDF/CSV.
- `SubscriptionsModule`: تجديد وإدارة الاشتراكات.
- `PlansModule`: إدارة خطط الـ Super Admin.
- `LicensesModule`: توليد السيريالات.
- `DistributorCalculatorModule`: حاسبة موزع الدحشة (مستقلة، Plug-in Architecture).
- `SuperAdminModule`: لوحة الإدارة العليا.

## 6. الإعداد للـ Distributor Calculator (Architecture Prep)
تم حجز مجلد `src/distributor-calculator/` في الهيكل. الوحدة ستكون:
- **مستقلة:** لا تؤثر على بقية الوحدات.
- **مرتبطة بالـ Tenant:** كل موزع ينتمي لشبكة عميل.
- **تستخدم العملة الديناميكية:** تأخذ `currency` من إعدادات الـ Tenant.
- **تظهر كتبويب منفصل** في الـ Bottom Navigation داخل Flutter.
