## مشروع "كروتك" / Kayansoft — نظام إدارة وتوزيع كروت الشبكات تلقائياً

| الحقل | القيمة |
|------|--------|
| اسم النظام | كروتك (Kurotek) — سابقاً "كروت الدحشة" (Dahsha Cards) |

| الإصدار | 1.0 (تحليل النظام القائم) |

## 1. مقدمة (Introduction)

### 1.1 الغرض من الوثيقة
توثيق شامل للنظام القائم (تطبيق أندرويد) واستخراج كامل المتطلبات الوظيفية وغير الوظيفية، ومنطق العمل، وهيكل البيانات، وتدفقات المعالجة، بحيث تصبح هذه الوثيقة المرجع المعتمد لإعادة بناء النظام على المعمارية الحديثة (Backend NestJS + PostgreSQL، وواجهة Flutter / أو ويب).

### 1.2 نطاق النظام (Scope)
"كروتك" نظام يُؤتمت بيع وتوزيع **كروت شحن شبكات الإنترنت (Wi‑Fi / MikroTik)** للعملاء في اليمن. تقوم الفكرة على:
1. استقبال رسائل SMS الواردة من **المحافظ المالية اليمنية** (جيب، جوالي، كريمي، ون كاش، إم فلوس، حاسب).
2. تحليل الرسالة آلياً لاستخراج (المبلغ + معرّف العميل + نوع المحفظة).
3. اختيار كارت شحن من المخزون مطابق لفئة المبلغ.
4. إرسال الكارت تلقائياً للعميل عبر SMS (أو بعد موافقة يدوية).
5. تسجيل كل عملية في سجلات الإيداعات والمعاملات والتقارير.

### 1.3 التعريفات والمصطلحات (Glossary)

| المصطلح | المعنى |
|---------|--------|
| الكارت (Card) | كود شحن شبكة Wi‑Fi، إما كود مفرد أو (اسم مستخدم + كلمة سر) لشبكات MikroTik |
| الفئة (Category) | قيمة الكارت بالريال اليمني (مثال: 100، 200، 250، 300، 500) |
| الإيداع (Deposit) | عملية دفع مالي وردت في رسالة SMS من محفظة |
| المعاملة (Transaction) | سجل أرشيفي لعملية شحن مكتملة (نجاح/فشل/رفض) |
| التفويض المعلّق (Pending Approval) | عملية تنتظر موافقة يدوية قبل إرسال الكارت |
| رمز الحساب (Account Code) | معرّف غير رقم هاتف مباشر (اسم/كود حساب) يستوجب مطابقة |
| مطابقة العميل (Customer Mapping) | ربط رمز حساب/اسم عميل برقم هاتفه الفعلي |
| الخدمة (Service) | المفتاح الرئيسي لتفعيل/تعطيل المعالجة التلقائية للرسائل |
| التوزيع التلقائي (Auto Send) | إرسال الكارت مباشرة عبر SmsManager دون تدخل بشري |

### 1.4 أصحاب المصلحة (Stakeholders)
- **صاحب المحل / مشغّل الشبكة (Operator):** المستخدم الأساسي الوحيد حالياً؛ يدير المخزون والإعدادات ويتابع العمليات.
- **العميل (Customer):** يدفع عبر المحفظة ويستلم كود الكارت عبر SMS. لا يتفاعل مع التطبيق مباشرة.
- **فريق الدعم / المالك التجاري:** يصدر أرقام التفعيل (السيريال).

## 2. الوصف العام للنظام القائم (Current System Overview)

## 3. تحليل ميزات النظام (Feature Analysis)

### 3.1 التفعيل والدخول
- شاشة تفعيل بـ **سيريال ثابت مضمّن في الكود** (`PY_7MD`) + اسم الشبكة.
- بعد التفعيل تُخزَّن حالة `is_activated` وتُعتبر جلسة دخول دائمة.
- يوجد كذلك "كلمة مرور تطبيق" افتراضية `PY_7MD` لإمكانية القفل/تغيير كلمة المرور.
- "تسجيل الخروج" يعيد ضبط `is_activated=false` ويعود لشاشة التفعيل.

### 3.2 إدارة المخزون (الكروت)
- إضافة الكروت بفئة محددة عبر:
  - **اللصق الجماعي (Bulk):** سطر لكل كارت، أو كتل مفصولة بنقطة `.`.
  - تنسيقات مدعومة: `user | pass`، `user / pass`، `المستخدم: X السر: Y`، أو كود مفرد.
  - **وضع تنسيق الكارت:** `user_pass` (مستخدم+سر) أو `user_only` (كود مفرد).
- استخراج الكروت من رسالة عبر **Gemini AI** (GeminiSmsAnalyzer) — تحليل نص واستخراج JSON بالكروت والفئة وتاريخ الصلاحية.
- عرض كل الكروت مع تصفية (الكل/فئة/متاحة/موزّعة)، نسخ، حذف فردي، حذف بالكامل.
- عدّادات حيّة للمخزون المتاح لكل فئة + إجمالي.
- **فئات ديناميكية:** إضافة/حذف فئات مخصصة (الافتراضي 100/200/250/300/500). حذف الفئة يحذف كروتها.

### 3.3 المعالجة التلقائية للرسائل (القلب التشغيلي)
- استقبال SMS في الخلفية حتى لو كان التطبيق مغلقاً.
- تحليل عبر Regex مخصص لكل محفظة + قوالب مخصصة من المستخدم.
- مفاتيح تحكم: تفعيل/تعطيل الخدمة كلياً، وتفعيل/تعطيل كل محفظة على حدة.
- وضعان للتشغيل:
  - **توزيع تلقائي (Auto Send):** إرسال فوري دون تدخل.
  - **موافقة يدوية:** إنشاء "تفويض معلّق" + إشعار بأزرار (موافقة/رفض).

### 3.4 التفويضات المعلّقة
- قائمة بالعمليات المنتظرة، مع إمكانية: الموافقة (إرسال الكارت)، الرفض (تسجيل إلغاء)، تعديل رقم العميل قبل الإرسال.
- أزرار الموافقة/الرفض متاحة أيضاً من داخل إشعار النظام.

### 3.5 العملاء الاستثنائيون (المطابقات)
- ربط (رمز حساب/اسم عميل) ← (رقم هاتف فعلي + اسم + محفظة).
- يُستخدم تلقائياً عند ورود دفعة برمز حساب بدلاً من رقم مباشر، لإرسال الكارت للرقم الصحيح.

### 3.6 التقارير
- تقارير الإيداعات والمعاملات مع تصفية حسب الفئة/المحفظة/الحالة.
- تصدير CSV (معاملات عامة + معاملات عميل محدد).
- مؤشرات وإحصاءات (Analytics) داخل التبويب.

### 3.7 الإعدادات
- الوضع الليلي/النهاري، اسم الشبكة، تغيير كلمة المرور.
- قوالب الرسائل: قالب عام، قالب رمز الحساب، رقم هاتف افتراضي لرمز الحساب.
- قوالب رسائل مخصصة معتمدة (Approved Templates) للتحليل التلقائي.
- مفتاح Gemini API مخصص.
- سلوك النقر على الإشعار (مشاركة عبر المؤلّف أم إشعار معلوماتي).

### 3.8 الإشعارات التفاعلية داخل التطبيق
- `NotificationBus` (SharedFlow) يطلق حدث `NewCardExtractedEvent` عند نجاح استخراج كارت.
- تلتقطه الواجهة الرئيسية وتعرض نافذة منبثقة: تفاصيل العملية + الكود + أزرار (مشاركة/فتح المحفظة، نسخ، إغلاق).
- فتح تطبيق المحفظة المعني (جيب/جوالي/ون كاش) ونسخ الكود تلقائياً.

## 4. منطق العمل التفصيلي (Business Logic)

### 4.1 دورة حياة الرسالة الواردة
1. `SmsReceiver` يستقبل `SMS_RECEIVED`. إذا كانت الخدمة معطّلة → تجاهل.
2. `SmsParser.parse()` يجرّب Regex لكل محفظة؛ وإن فشل يجرّب القوالب المخصصة.
3. عند النجاح: استخراج (amount, identifier, walletType, isAccountCode).
4. فحص تفعيل المحفظة المعنية؛ إن كانت معطّلة → تجاهل.
5. تحديد المستلم: مطابقة `customer_mappings` أولاً ← ثم (رقم افتراضي لرمز الحساب) ← ثم المعرّف نفسه.
6. **إذا التوزيع التلقائي مفعّل:**
   - جلب كارت غير مستخدم بالفئة → إن وُجد: وسمه مستخدماً، صياغة الرسالة، إرسال SMS، تسجيل Transaction + Deposit (`isShared = isSent`)، إطلاق حدث NotificationBus، إشعار نجاح/فشل.
   - إن لم يوجد كارت (نفاد المخزون): إرسال اعتذار، تسجيل فشل، إشعار "نفذ المخزن".
7. **إذا الوضع يدوي:** إنشاء Deposit (معلّق) ثم PendingApproval مرتبط به، وإظهار إشعار بأزرار موافقة/رفض.

### 4.2 منطق تحديد "رقم هاتف مباشر" مقابل "رمز حساب"
- يُعتبر رقم هاتف مباشراً إذا كان طوله 9 أرقام وكلها أرقام؛ غير ذلك يُعامل كرمز حساب (`isAccountCode=true`) يستلزم مطابقة.

### 4.3 منطق الموافقة اليدوية
- عند الموافقة: مطابقة العميل (إن وُجد)، جلب كارت بالفئة، الإرسال، تسجيل المعاملة وتحديث الإيداع وحذف التفويض. عند نفاد المخزون: رسالة اعتذار وتسجيل فشل.
- عند الرفض: تسجيل معاملة إلغاء، تحديث الإيداع كـ"مرفوض"، حذف التفويض.

### 4.4 صياغة نص الكارت
- إن وُجدت كلمة سر: نص رأسي `اسم المستخدم:\n... كلمة السر:\n...`، وإلا يُرسل `code` المفرد.
- قوالب الرسائل تدعم متغيرات: `%amount` `%phone` `%account` `%code` `%wallet` `%category` `%details`.

## 7. تدفق البيانات (Data Flow)

```
SMS → SmsReceiver → SmsParser → CardRepository(Room) ──► StateFlow ──► ViewModel ──► Compose UI
                                       │
                                       ├─► SmsSender (SmsManager) → عميل
                                       ├─► NotificationManager (إشعارات نظام)
                                       └─► NotificationBus (SharedFlow) → نافذة داخل التطبيق

UI Actions (إضافة كروت/موافقة/إعدادات) → ViewModel → Repository → Room/SharedPreferences → StateFlow → UI
```
- مصدر الحقيقة الوحيد: `CardRepository`. كل القراءات تفاعلية عبر `Flow/StateFlow` فتتحدث الواجهة تلقائياً.

## 8. تحليل نظام قراءة الرسائل (SMS Parsing Engine)

محرّك Regex مخصص لكل محفظة داخل `SmsParser`:

| المحفظة | النمط النموذجي | المخرجات |
|---------|----------------|----------|
| جيب | `500ر.ي ... -777777777` | المبلغ + المعرّف |
| جوالي | `استلمت مبلغ 5100 YER من 700070034 ...` | المبلغ + الرقم |
| كريمي | `أودع/الاسم لحسابك مبلغ 7400 ...` | الاسم(حساب) + المبلغ |
| ون كاش | `استلمت 200.00 من الاسم رصيدك...` | المبلغ + الاسم(حساب) |
| إم فلوس | `تم إيداع مبلغ 3000 YER من: الاسم المرجع:...` | المبلغ + الاسم(حساب) |
| حاسب | `اضيف 200ر.ي تحويل مشترك ... من 120025` | المبلغ + المعرّف(حساب) |
| قوالب مخصصة | يكتبها المستخدم بـ `%amount/%phone/%account` | تتحوّل إلى Regex ديناميكي |

**تحليل AI:** `GeminiSmsAnalyzer` يرسل النص إلى `generativelanguage.googleapis.com` (نموذج معرّف كـ `gemini-3.5-flash`) لاستخراج الكروت كـ JSON — يُستخدم لاستيراد الكروت من رسائل المورّد لا لتحليل الإيداعات.

## 9. تحليل نظام توزيع الكروت (Card Distribution)

- **اختيار الكارت:** `getUnusedCardByCategory` يأخذ أول كارت `used=0` للفئة (FIFO تقريبي حسب id).
- **الوسم:** `markCardAsUsed` فور الاختيار لمنع التكرار.
- **الإرسال:** `SmsSender.sendSmsInBackground` عبر `SmsManager`، مع رفض الإرسال إذا كان المستلم اسماً/نصاً (يحتوي حروفاً) → في هذه الحالة يُعرض إشعار "مشاركة يدوية".
- **فتح المحفظة:** `launchWalletApp` ينسخ الكود ويفتح تطبيق المحفظة (حزم: جيب/جوالي/ون كاش) أو ورقة مشاركة عامة.
- **التسجيل المزدوج:** كل توزيع يُنشئ Deposit + Transaction، وحالة `isShared` تعكس نجاح الإرسال الفعلي.

## 10. المتطلبات الوظيفية (Functional Requirements)

| المعرّف | المتطلب |

## 12. اكتشاف المشاكل ونقاط الضعف (Findings & Risks)

### 12.1 أمنية
- **سيريال تفعيل ثابت مضمّن** (`PY_7MD`) وكذلك كلمة المرور الافتراضية → يسهل تجاوز الحماية. لا يوجد تحقق خادمي حقيقي.
- لا يوجد تشفير للبيانات الحساسة (أكواد الكروت) محلياً.
- مفتاح Gemini قد يُخزَّن نصّاً صريحاً في التفضيلات.

### 12.2 سلامة البيانات
- `fallbackToDestructiveMigration` → ترقية المخطط تمسح كل البيانات (خطر فقدان مخزون/سجلات).
- غياب قيود FK وفهارس → اعتماد على منطق التطبيق فقط لتماسك البيانات.
- ربط `depositId` منطقي بلا ضمان مرجعي.

### 12.3 موثوقية المعالجة
- الاعتماد على Regex صارم؛ أي تغيير بسيط في صيغ رسائل المحافظ يكسر التحليل.
- منطق "9 أرقام = هاتف" هشّ (أرقام بصيغ مختلفة قد تُصنَّف خطأ).
- لا يوجد منع ازدواجية لرسالة واحدة قد تصل مرتين (لا فحص تكرار/Idempotency).
- إرسال SMS قد يفشل صامتاً (حدود المشغّل/الرصيد) دون آلية إعادة محاولة.

### 12.4 معماري/تشغيلي
- منطق عمل مكرّر في ثلاثة مواضع (SmsReceiver، PendingApprovalReceiver، ViewModel) → صعوبة صيانة وتعارض محتمل (خرق DRY الوارد في `PROJECT_RULES.md`).
- شاشة `MainDashboardScreen` ضخمة جداً (تجمع كل التبويبات والمنطق).
- نظام أحادي الجهاز بلا نسخ احتياطي سحابي ولا تعدد مستخدمين/فروع.
- لا اختبارات وظيفية حقيقية تغطي محرك التحليل والتوزيع.

### 12.5 امتثال
- صلاحيات SEND/READ/RECEIVE SMS تخضع لقيود Google Play الصارمة وقد تمنع النشر دون مبرر مؤهَّل.

## 13. فجوة الانتقال إلى المعمارية المستهدفة (Gap Analysis)

وفق `PROJECT_RULES.md`، النظام المستهدف: **NestJS + PostgreSQL + Prisma + JWT (Backend)** و**Flutter (Clean Architecture)**. أبرز الفجوات الواجب حسمها قبل التصميم:

1. **أين يبقى استقبال SMS؟** قراءة SMS قدرة خاصة بجهاز أندرويد؛ الخادم لا يقرأ رسائل المحفظة. يلزم تحديد: هل يبقى تطبيق أندرويد "وكيلاً" (Agent) يلتقط الرسائل ويرسلها للـ API؟ أم يُعتمد على Webhooks من مزوّد المحافظ (غالباً غير متاح)؟
2. **نموذج التعددية:** خادم مركزي يفتح الباب لتعدد المحلات/المستخدمين/الفروع وإدارة مخزون مركزي.
3. **الأمان:** استبدال السيريال الثابت بنظام مصادقة JWT وأدوار وصلاحيات.
4. **سلامة البيانات:** PostgreSQL + Prisma Migrations + قيود FK وفهارس وIdempotency.
5. **الواجهة:** Flutter بمعمارية Feature-First مع الحفاظ على هوية وتجربة الاستخدام الحالية + الاستفادة من نموذج `Disgain`.
